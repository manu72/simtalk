# Architecture Map

Generated by docsync on 2026-05-21.

## Summary

SimTalk is a pnpm TypeScript monorepo for a private Phase 1 speech-to-speech translation prototype. The browser owns UI state, microphone capture, WebRTC negotiation with OpenAI, translated audio playback, and transcript display. The backend is a small Node/Hono API that issues short-lived OpenAI Realtime Translate client secrets and enforces validation, CORS, rate limiting, and baseline security headers. Shared Zod schemas and TypeScript types define the API contract between the frontend and backend.

Phase 1 has no database, no server-side audio/transcript storage, and no server-side audio proxy. The browser talks to OpenAI directly over WebRTC after receiving a short-lived client secret from the backend.

## Runtime Flow

1. User opens the React/Vite app.
2. User selects a Phase 1 mode and target language. Turn-about and Practice also include a source language.
3. Frontend validates and sends a request to `POST /realtime/token`.
4. Backend validates the request with shared Zod schemas.
5. Backend rate-limits the request per client IP and calls OpenAI's realtime translation client-secret endpoint using server-side `OPENAI_API_KEY`.
6. Backend returns a browser-safe token response with `clientSecret`, session expiry fields, session id, and the OpenAI translation calls URL.
7. User explicitly starts microphone/WebRTC.
8. Browser captures microphone audio, creates an SDP offer, and posts the offer directly to OpenAI's translation calls endpoint using the short-lived client secret.
9. Browser receives remote translated audio and transcript deltas from OpenAI.
10. Browser stops and cleans up local tracks, data channel, peer connection, and audio element when the user stops audio, switches mode, startup is aborted, or the app unmounts.

## Primary Languages And Frameworks

- TypeScript across runtime packages.
- React 19 and Vite 7 for the frontend.
- Hono on Node.js for the backend.
- Zod for shared runtime validation and inferred TypeScript types.
- Vitest for backend, frontend, shared, and Agentic OS script tests.
- React Testing Library and jsdom for frontend component tests.
- Playwright for E2E smoke tests.
- Python standard library scripts under `scripts/agentic/` for Agentic OS memory and routing.

## Entrypoints

- Frontend dev/build config: `frontend/vite.config.ts`.
- Backend server: `backend/src/server.ts`.
- Backend app factory: `backend/src/app.ts`.
- Backend routes:
  - `GET /health` from `backend/src/routes/health.ts`.
  - `POST /realtime/token` from `backend/src/routes/realtime.ts`.
- Shared contract entrypoint: `shared/types/src/index.ts`.
- E2E config: `playwright.config.ts`.

## Storage, Messaging, And Background Work

- Database: none in Phase 1.
- Server-side audio/transcript storage: none.
- Browser local recording: planned, not implemented yet.
- Messaging/job systems: none.
- Realtime transport:
  - Browser to backend: HTTP JSON for token issuance.
  - Browser to OpenAI: WebRTC SDP exchange, remote audio, and data-channel transcript events.
  - Backend to OpenAI: server-side HTTP request to mint the short-lived client secret.

## Security And Privacy Boundaries

- `OPENAI_API_KEY` is read only by the backend and never returned to the browser.
- Browser receives only a short-lived OpenAI client secret.
- Backend does not receive, proxy, log, or store audio/transcript content.
- CORS only reflects configured allowed origins.
- `POST /realtime/token` is protected by an in-memory per-client rate limiter.
- Security headers are applied globally by backend middleware:
  - Content-Security-Policy
  - Strict-Transport-Security
  - X-Frame-Options
  - Referrer-Policy
  - X-Content-Type-Options

## Deployment And Runtime Targets

- Current in-repo deployment config: none.
- Planned Phase 1 target from architecture docs and memory: Vercel, protected by Vercel Password Protection and custom domain `simtalk.app`.
- Planned Phase 2 target: Google Cloud Run plus real auth, persistence, and a room/media layer.
- No `.github/workflows` directory or Dockerfile exists.

## Test Strategy

- Root scripts orchestrate package-level build, typecheck, unit tests, and E2E.
- Backend Vitest tests cover config parsing, health/CORS/security headers, realtime token validation, OpenAI error mapping, and rate limiting.
- Frontend Vitest/RTL tests cover product shell rendering, accessible mode controls, token requests, stale response handling, explicit WebRTC start, transcript rendering, and cleanup.
- Shared Vitest tests cover conversation mode, language, token, error, and health schemas.
- Playwright E2E currently smoke-tests the rendered shell and mode controls against the frontend dev server only.
- Agentic OS Python scripts have unittest coverage under `scripts/agentic/test_*.py`.

## Open Questions And Uncertainties

- In-repo deploy wiring is not present; Vercel settings are out-of-tree.
- `SESSION_SECRET` and `VERCEL_PROTECTION_BYPASS_SECRET` are documented in env examples but not used by current backend code.
- Browser-local recording and downloads are product requirements but are not implemented yet.
- Mode-specific Turn-about and Practice behavior is still pending beyond controls and source-language handling.
- Observability is limited to current tests and console/server output; no metrics provider is configured.
