# Architecture And Tech Stack Map

Generated by docsync on 2026-05-24.

## Architecture Summary

SimTalk is a TypeScript pnpm workspace for a private, pre-MVP realtime speech-to-speech translation prototype. The current codebase spans Phase 1 local modes and Phase 1.5 two-person remote rooms.

Runtime boundaries:

1. Browser app (`frontend/`)
   - Renders the lobby, local mode surfaces, remote room surface, transcript sheet, developer drawer, and summary surface.
   - Requests short-lived OpenAI realtime translation credentials from the backend.
   - Requests LiveKit room creation and room participant tokens from the backend for remote rooms.
   - Captures microphone audio after user action.
   - Opens a direct WebRTC session with OpenAI using a browser-safe client secret.
   - For remote rooms, connects to LiveKit, publishes the local microphone, subscribes to the remote participant microphone, and feeds that remote track into the OpenAI translation session.
   - Plays translated audio, optionally mutes/unmutes original remote audio, renders transcript deltas, manages mode-specific session state, records local browser audio where implemented, and tears down media resources.

2. Hono backend (`backend/`)
   - Exposes `GET /health`, `POST /realtime/token`, `POST /rooms`, and `POST /rooms/:roomId/token`.
   - Validates requests with shared Zod schemas from `@simtalk/shared-types`.
   - Reads `OPENAI_API_KEY`, `LIVEKIT_API_KEY`, and `LIVEKIT_API_SECRET` server-side only.
   - Calls OpenAI's realtime translation client-secret endpoint.
   - Creates LiveKit rooms and signs room-scoped participant JWTs.
   - Applies CORS, security headers, sanitized error mapping, `Cache-Control: no-store`, and in-memory rate limiting.
   - Does not proxy, store, or inspect audio/transcript content.

3. Vercel API adapter (`api/`)
   - `api/index.ts` imports `createApp()` from backend source and mounts it under `/api`.
   - `vercel.json` rewrites `/api/(.*)` to the adapter and all non-API paths to the Vite SPA.

4. OpenAI Realtime Translate
   - Uses `gpt-realtime-translate` through `/v1/realtime/translations`.
   - Accepts browser SDP offers at the translations calls endpoint.
   - Handles speech recognition, translation, synthesized translated audio, and transcript deltas.
   - Public docs describe dynamic voice adaptation for general tone, pitch, and style. They do not promise exact voice cloning or guaranteed speaker-identity/gender matching.

5. LiveKit Cloud
   - Provides the remote-room media layer for Phase 1.5.
   - The backend creates two-participant rooms and participant tokens.
   - The browser connects directly to LiveKit with the short-lived participant token.

## Runtime Flows

Local OpenAI translation flow:

```text
Browser UI
  -> POST /realtime/token
  -> Hono validates request and applies rate limit
  -> Hono calls OpenAI client-secret endpoint with OPENAI_API_KEY
  -> Hono returns short-lived client secret and translation calls URL
  -> Browser captures microphone and sends SDP offer directly to OpenAI
  -> OpenAI returns SDP answer, translated audio, and transcript deltas
  -> Browser renders/transcribes/records locally and cleans up on stop
```

Remote room flow:

```text
Browser lobby
  -> POST /rooms
  -> Hono creates a LiveKit room and returns /rooms/:roomId
  -> Browser navigates to the room URL
  -> POST /rooms/:roomId/token
  -> Hono signs a short-lived LiveKit participant token
  -> Browser connects directly to LiveKit and publishes local mic
  -> Browser subscribes to remote mic track
  -> Browser requests an OpenAI realtime token and translates that remote track
```

## Entrypoints

- `frontend/src/main.tsx` renders the React app and imports `frontend/src/styles/tokens.css`.
- `frontend/src/App.tsx` is the frontend session orchestrator and pathname-based room router.
- `frontend/src/realtimeTokenClient.ts` owns browser-to-backend realtime token requests and client-side schema/error validation.
- `frontend/src/roomTokenClient.ts` owns browser-to-backend room create/join requests.
- `frontend/src/realtimeTranslationSession.ts` owns microphone or supplied-stream WebRTC offer/answer exchange, OpenAI data-channel transcript deltas, remote audio attachment, and teardown.
- `frontend/src/liveKitRemoteRoomSession.ts` owns LiveKit connect/publish/subscribe behavior and bridges remote audio into OpenAI translation.
- `backend/src/server.ts` starts the Hono server with `@hono/node-server` for local development.
- `backend/src/app.ts` wires middleware, routes, and shared error handling.
- `api/index.ts` mounts the Hono app under `/api` for Vercel.
- `shared/types/src/index.ts` is the cross-boundary contract hub.

## Storage, Messaging, And Jobs

- No database exists in Phase 1 / 1.5.
- No server-side persistence of audio, transcripts, or PII exists.
- No queue, worker, cache, object storage, ORM, or migration system exists.
- Browser-local transcript/audio downloads are implemented for current session artifacts only.
- Remote-room reload continuity stores only an opaque participant identity in `sessionStorage`.

## Tests And CI

- Tests live under root `tests/`, not colocated with packages.
- Per-package Vitest configs point at root test folders.
- Playwright starts the frontend only and mocks token/OpenAI/room network boundaries.
- `.github/workflows/ci.yml` runs formatting check, lint, typecheck, Vitest suites, build, and Chromium Playwright tests.
- `.github/workflows/vercel-deploy.yml` runs after successful CI workflow runs and deploys previews for non-`main` pushes and production for `main`.

## Uncertainties

- Manual live OpenAI validation is still pending.
- Manual live LiveKit room validation is still pending.
- The backend validates `mode` and `sourceLanguage`, but currently forwards only `targetLanguage` and transcription model settings to OpenAI for token minting.
- `roomTokenRequestSchema` validates `sourceLanguage` and `targetLanguage`; the backend uses the room identity/display name for JWT minting while the frontend uses the language request to configure the translation leg.
- Agentic subsystem notes, `PRD.md`, and parts of `CLAUDE.md` contain stale planned-status or Phase 2 room claims; README and these generated docsync artifacts should be treated as the more current repository documentation after this pass.
